22. **Task - Release v1.0.22** (v1.0.22): cert SANs go exact-name; certs/README.md explains why<br>
    **Result**: Investigation of v1.0.21 found OpenSSL 3.x rejects single-label wildcards (`*.localhost`) as TLD-like — `openssl x509 -checkhost hub.localhost` against `*.localhost` SAN returns "does NOT match", and curl/Python ssl behave the same. The v1.0.21 baseline was therefore broken for `jupyterhub.localhost` access. Replaced `cert_dns_altnames` with enumerated SANs covering exactly the surfaces the deployment exposes: `<sub>.<base>`, `<base>`, `traefik.<base>`, plus the localhost-side equivalents (`<sub>.localhost`, `localhost`, `traefik.localhost`). New `template/certs/README.md` documents the OpenSSL strict rule, the multi-label DNS wildcard that DOES work, the `*.app.localhost` fake-intermediate-label workaround, and the `openssl x509 -checkhost` verification recipe. CI matrix scenarios updated (no more `*.` prefix in any SAN; cert folder name is now the literal first SAN). All three primary scenarios verified strict-OpenSSL-clean for every covered hostname. Bumped 1.0.21 -> 1.0.22, tagged, pushed.
